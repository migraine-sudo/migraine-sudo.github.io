picotcp Integer overflow

发表于1899-11-30|更新于2021-08-25
|阅读量:

Author:xiaolinfeng

I perform code audit on picotcp, and picotcp will parse the received DNS packets.

The basic unit of DNS request and response is the DNS message (Message). The DNS message structure of the request and response is exactly the same, and each message is composed of the following five sections:

DNS Header is a part of every DNS message, and its length is fixed at 12 bytes. The Question part stores the domain name data that is queried from the server. Generally, it has only one Entry. The format of each Entry is the same, as shown below:

QNAME is a domain name composed of a sequence of labels. The format of QNAME uses DNS standard name notation. This field is variable length, so there may be an odd number of bytes, but it is not filled. DNS uses a standard format to encode domain names. It consists of a series of labels (different from the labels separated by. In the domain name). The upper two bits of the first byte of each label are used to indicate the type of label. Two of the four are allocated in RFC1035, namely: the ordinary label represented by 00 and the compressed label represented by 11 (0xC0).

The vulnerability comes from the pico_dns_qname_to_url function，memcpy((void *)url, (void *)(temp + 1), (size_t)(namelen - 1));When the length of the input qname is less than 1 (controllable), resulting in an integer overflow, memcpy will write an excessive amount of data to the stack.

It is easy to write a poc (exploit is still difficult, depending on how the person who uses this protocol stack writes application layer code for the protocol stack)

The poc is as follows, which can generate an integer overflow. In actual writing and utilization, you only need to modify the qname of the DNS to \x0.
